# ВВЕДЕНИЕ

## Актуальность темы

В современном мире цифровые финансовые технологии проникают во все сферы экономической деятельности. Криптовалюты, в частности стейблкоины (stablecoins), такие как USDT (Tether), стали полноценным инструментом для международных расчётов, хеджирования валютных рисков и P2P-торговли. В условиях ограниченного доступа граждан РФ к традиционным зарубежным платёжным системам, P2P-обмен (peer-to-peer, «от человека к человеку») приобрёл особую значимость как альтернативный канал конвертации рублей в цифровые активы и обратно.

Однако организация P2P-обмена вручную сопряжена с целым рядом проблем:
- **Высокая операционная нагрузка** на посредников (операторов), которым приходится вручную сопоставлять заявки, пересылать реквизиты, отслеживать сроки и фиксировать статусы сделок;
- **Риск человеческой ошибки**: ошибочные суммы, неверные реквизиты, пропуск сроков оплаты;
- **Отсутствие централизованного аудита**: данные о сделках разбросаны по перепискам в мессенджерах, что затрудняет учёт, налогообложение и разрешение спорных ситуаций;
- **Низкая масштабируемость**: ручной процесс не позволяет обслуживать более 5–10 сделок одновременно без снижения качества.

Информационные системы на базе чат-ботов в мессенджерах (Telegram, WhatsApp, VK) демонстрируют высокую эффективность в задачах автоматизации рутинных операций: бот не устаёт, работает 24/7, ведёт структурированный журнал событий и позволяет интегрировать внешние источники данных (API бирж, Центробанка) для актуализации справочной информации.

Таким образом, разработка специализированной информационной системы с чат-ботом для автоматизации P2P-обмена USDT на рубли является актуальной задачей, позволяющей повысить операционную эффективность, снизить риски ошибок и обеспечить прозрачность ведения сделок.

# ГЛАВА 1. АНАЛИТИЧЕСКАЯ ЧАСТЬ

## 1.1. Анализ предметной области peer-to-peer обмена цифровых валют

### 1.1.1. Понятие P2P-обмена и его место в цифровой экономике

Peer-to-peer (P2P) обмен — это форма торговли цифровыми активами, при которой сделка происходит непосредственно между двумя пользователями (физическими или юридическими лицами) без посредничества централизованной биржи в части хранения средств. Покупатель перечисляет фиатную валюту (в данном случае — российские рубли) на реквизиты продавца, а продавец, получив подтверждение платежа, отправляет эквивалентную сумму в криптовалюте (USDT) на кошелёк покупателя.

P2P-площадки (Binance P2P, Bybit P2P, KuCoin P2P, LocalBitcoins) выполняют функцию escrow-агента: замораживают криптовалюту продавца до момента подтверждения получения фиата, что снижает риск мошенничества. Однако в условиях санкционных ограничений российские пользователи сталкиваются с блокировкой доступа к ряду крупных площадок, а также с ограничениями на работу с российскими банками.

В этих условиях возникает ниша **локальных P2P-сервисов**, организованных через закрытые Telegram-каналы и чат-боты. Такие сервисы работают по принципу «тёмного пула ликвидности»: трейдеры (продавцы) предоставляют свои банковские реквизиты, операторы (менеджеры) создают заявки от покупателей, а владелец сервиса контролирует общий процесс и владеет административными функциями.

### 1.1.2. Жизненный цикл P2P-сделки

Классический жизненный цикл сделки USDT/RUB включает следующие этапы:

1. **Инициация**: оператор получает запрос от покупателя на обмен суммы X рублей в USDT.
2. **Расчёт**: система делит сумму в рублях на текущий курс USDT/RUB и получает эквивалент в USDT.
3. **Подбор реквизитов**: система выбирает активного трейдера, чьи лимиты (min–max) перекрывают запрашиваемую сумму.
4. **Выдача реквизитов**: покупателю отправляются банковские реквизиты (номер карты или телефон для СБП) выбранного трейдера.
5. **Оплата**: покупатель переводит рубли в течение установленного таймаута (обычно 15–30 минут).
6. **Подтверждение**: трейдер, получив уведомление от банка, подтверждает поступление средств в системе.
7. **Завершение**: сделка получает статус «подтверждена», трейдер отправляет USDT покупателю вне системы (через блокчейн), а система фиксирует статус для учёта.

На любом этапе может возникнуть конфликт (спор): покупатель утверждает, что перевёл деньги, а трейдер — что не получил. В таком случае необходим механизм разрешения спора с участием владельца.

### 1.1.3. Управление курсом валюты

Актуальный курс USDT/RUB — критически важный параметр для любого обменного сервиса. Курс формируется под влиянием нескольких факторов:
- Курс доллара США к рублю, устанавливаемый ЦБ РФ;
- Ликвидность и спред на крупных P2P-площадках (Binance, Bybit, Huobi);
- Премия/дисконт в зависимости от способа оплаты (СБП обычно дешевле, чем перевод по номеру карты);
- Региональные ограничения и спрос.

Для обеспечения конкурентоспособности сервис должен либо получать курс в реальном времени из открытых API, либо позволять владельцу устанавливать его вручную. Надёжная практика — **многоисточниковый парсинг** с fallback на резервное значение при недоступности внешних сервисов.

## 1.2. Обзор существующих решений для автоматизации P2P-обмена

### 1.2.1. Централизованные P2P-площадки

**Binance P2P** — крупнейшая в мире P2P-площадка, интегрированная в экосистему Binance. Предоставляет API для получения актуальных объявлений (ads), escrow, репутационную систему. Минус: ограниченный доступ для резидентов РФ, требование KYC.

**Bybit P2P** — аналогичная площадка с поддержкой рублевых пар. Имеет REST API для получения тикеров spot-рынка. P2P-API имеет ограниченную документацию.

**KuCoin P2P** — поддерживает рублёвые сделки через сторонние платёжные системы. Предоставляет API для списка объявлений.

Общий недостаток централизованных площадок: жёсткая верификация, риск заморозки средств, зависимость от политики площадки.

### 1.2.2. Чат-боты и локальные сервисы

На российском рынке существуют десятки Telegram-ботов для P2P-обмена. Они условно делятся на две категории:

**Простые боты-обменники** — работают по принципу «введите сумму → получите реквизиты → оплатите». Не хранят историю, не имеют ролевой модели, требуют ручного вмешательства администратора на каждой сделке.

**Боты с базой данных** — используют простые SQLite-хранилища, позволяют вести журнал заявок. Часто имеют одну роль (админ–клиент) и не разграничивают прав операторов и трейдеров.

### 1.2.3. Сравнительный анализ и выявление недостатков

| Критерий | Централизованные площадки | Простые боты | Требуемое решение |
|----------|---------------------------|--------------|-------------------|
| Автоматизация | Высокая | Низкая | Высокая |
| Ролевая модель | Есть (KYC-роли) | Отсутствует | Owner/Operator/Trader |
| Хранение истории | Полное | Частичное | Полное с мягким удалением |
| Автообновление курса | Есть | Отсутствует | Многоисточниковый парсинг |
| Спорная система | Есть | Отсутствует | Полный цикл спора |
| Депозиты/страховки | Есть | Отсутствует | Страховой + рабочий депозит |
| Облачное развёртывание | Собственные серверы | Локальный запуск | PaaS (Render) |

**Вывод**: ни одно из существующих решений не покрывает специфические потребности локальных P2P-команд, работающих в условиях ограниченного доступа к крупным площадкам. Необходимо разработать собственную систему, объединяющую автоматизацию чат-бота, ролевой доступ, аудит сделок и многоисточниковый парсинг курса.

## 1.3. Формирование требований к информационной системе

### 1.3.1. Функциональные требования

**F1. Авторизация и аутентификация:**
- Доступ к системе осуществляется по одноразовым токенам длиной 16 символов.
- Токены генерируются владельцем и привязываются к роли (owner, operator, trader).
- Токены хранятся в защищённом виде в БД с флагом активности.

**F2. Ролевая модель:**
- **Владелец (Owner)** — полный доступ: управление токенами, курсом, пользователями, статистика, удаление сделок, подтверждение депозитов.
- **Оператор (Operator)** — создание заявок, просмотр активных заявок, поиск по номеру/сумме, открытие споров.
- **Трейдер (Trader)** — управление реквизитами, подтверждение/отклонение сделок, просмотр споров, пополнение депозита.

**F3. Управление сделками:**
- Оператор создаёт сделку, указав сумму в RUB.
- Система автоматически подбирает активного трейдера с подходящими лимитами.
- Система рассчитывает эквивалент в USDT по текущему курсу.
- Сделке присваивается уникальный номер и устанавливается таймаут (30 минут).
- Трейдер получает уведомление и может подтвердить или отклонить сделку.

**F4. Управление реквизитами:**
- Трейдер добавляет реквизиты двух типов: банковская карта и СБП.
- Для каждого реквизита задаются: название, банк, ФИО, номер карты/телефон, минимальная и максимальная сумма.
- Реквизиты можно активировать/деактивировать и удалять.

**F5. Автоматическое обновление курса:**
- Система опрашивает три источника: Binance P2P API, KuCoin P2P API, Центральный банк РФ (USD + 2% премия).
- При успешном получении курса (диапазон 50–200 RUB/USDT) он сохраняется в БД.
- При недоступности всех источников используется резервный курс 78.50.
- Обновление происходит каждый час по расписанию.

**F6. Система споров:**
- Оператор может открыть спор по активной сделке.
- Спор фиксируется в таблице `disputes` со связью к сделке.
- Владелец просматривает спорные сделки и принимает решение.

**F7. Депозитная система:**
- Каждый трейдер обязан внести страховой депозит (1000 USDT) для допуска к работе.
- Владелец подтверждает поступление депозита и активирует аккаунт.
- Трейдер может пополнять рабочий депозит для увеличения лимитов.

**F8. Статистика и аналитика:**
- Владелец видит общую статистику по пользователям, сделкам, оборотам, статусам.
- Оператор видит статистику только по своим заявкам.
- Трейдер видит статистику по своим сделкам в личном кабинете.

### 1.3.2. Нефункциональные требования

**NF1. Производительность:** система должна обрабатывать до 100 одновременных подключений без потери отзывчивости.

**NF2. Надёжность:** при сбое внешнего API курса валют система должна продолжать работу с резервным курсом.

**NF3. Безопасность:** токены хранятся в открытом виде (так как это разовые пригласительные ключи), но доступ к БД ограничен. Используется webhook-архитектура с HTTPS.

**NF4. Масштабируемость:** архитектура должна позволять подключение дополнительных обработчиков (handlers) и сервисов без изменения ядра.

**NF5. Доступность:** система развёртывается на облачной платформе Render с keep-alive-механизмом, предотвращающим засыпание бесплатного инстанса.

### 1.3.3. Требования к технологическому стеку

- Язык программирования: Python 3.11+

- Планировщик задач: APScheduler (AsyncIOScheduler)
- Парсинг HTML/JSON: BeautifulSoup4 + aiohttp

- Развёртывание: Render (PaaS)

# ГЛАВА 2. ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ

## 2.1. Обоснование выбора технологий разработки

### 2.1.1. Язык программирования Python

Python выбран в качестве основного языка разработки по следующим причинам:
- **Высокая скорость разработки**: лаконичный синтаксис, обширная стандартная библиотека и огромное количество готовых пакетов снижают time-to-market.
- **Экосистема для Telegram-ботов**: библиотека aiogram является де-факто стандартом для разработки асинхронных ботов на Python и активно поддерживается сообществом.
- **Асинхронность**: нативная поддержка `async/await` (начиная с Python 3.5+) позволяет эффективно обрабатывать I/O-bound операции (HTTP-запросы, запросы к БД) в одном потоке, что критично для чат-бота с множеством одновременных пользователей.
- **Читаемость кода**: синтаксис Python близок к псевдокоду, что упрощает сопровождение проекта и вовлечение новых разработчиков.

### 2.1.2. Фреймворк aiogram 3.x

Aiogram — это современный асинхронный фреймворк для работы с Telegram Bot API. Версия 3.x предоставляет ряд архитектурных улучшений по сравнению с 2.x:

- **Router-based архитектура**: обработчики (handlers) группируются в Router'ы, что позволяет разделять логику по модулям (common, operator, trader, owner) и легко подключать их к Dispatcher.
- **FSM (Finite State Machine)**: встроенная поддержка конечных автоматов для реализации многошаговых диалогов (например, добавление реквизита: название → тип → банк → ФИО → номер → лимиты).
- **Фильтрация**: декларативная система фильтров (`F.text == "..."`, `F.data.startswith(...)`) делает код обработчиков чистым и понятным.
- **Webhook и polling**: поддержка обоих режимов получения обновлений. Для облачного развёртывания выбран webhook.

### 2.1.3. СУБД PostgreSQL и asyncpg

PostgreSQL выбрана по причинам:
- **Надёжность ACID**: гарантии целостности данных при одновременном доступе нескольких пользователей.
- **Расширенные типы данных**: JSONB, DECIMAL с точностью до 15 знаков (важно для финансовых расчётов).
- **Управление пулом соединений**: `asyncpg` предоставляет высокопроизводительный асинхронный драйвер с пулом соединений, что критично для бота с множеством одновременных запросов.

### 2.1.4. Парсинг и HTTP-клиент

Для получения курса валют используется `aiohttp` — асинхронный HTTP-клиент/сервер, интегрированный с `asyncio`. Он позволяет выполнять параллельные запросы к нескольким API без блокировки event loop.

`BeautifulSoup4` используется как резервный инструмент для разбора HTML, однако в текущей реализации все источники курса предоставляют JSON-API, поэтому BS4 импортирован на случай будущих расширений.

### 2.2.3. Механизм взаимодействия с Telegram

Выбрана **webhook-архитектура** вместо polling по следующим причинам:
- **Эффективность**: сервер получает обновления по инициативе Telegram, не расходуя ресурсы на постоянные HTTP-запросы.
- **Совместимость с PaaS**: платформа Render требует, чтобы приложение слушало HTTP-порт, что естественно для webhook-сервера.
- **Масштабируемость**: при необходимости можно добавить балансировщик нагрузки и несколько инстансов бота.

Жизненный цикл webhook-запроса:
1. Пользователь отправляет сообщение в Telegram.
2. Telegram Bot API пересылает JSON-обновление на URL `https://trade-prog-2.onrender.com/webhook`.
3. aiohttp-сервер принимает запрос и передаёт его в `SimpleRequestHandler` aiogram.
4. Dispatcher направляет обновление в соответствующий Router и handler.
5. Handler выполняет бизнес-логику (запросы к БД, отправку ответов).
6. Бот отправляет ответ пользователю через Telegram Bot API.

### 2.2.4. Keep-alive механизм

Платформа Render в бесплатном тарифе останавливает инстанс через 15 минут неактивности. Для предотвращения этого реализована фоновая задача `keep_alive_ping`, которая каждые 10 минут выполняет HTTP-GET запрос на эндпоинт `/health` собственного сервера. Это поддерживает процесс в активном состоянии.

## 2.3. Проектирование базы данных

### 2.3.1. Логическая модель данных

База данных спроектирована в третьей нормальной форме (3NF) и включает шесть взаимосвязанных сущностей:

**1. users** — Пользователи системы
- `user_id` (BIGSERIAL, PK) — внутренний идентификатор
- `telegram_id` (BIGINT, UNIQUE) — ID в Telegram
- `username` (VARCHAR) — публичное имя
- `role` (VARCHAR) — роль: owner, operator, trader
- `token` (VARCHAR) — текущий токен авторизации
- `insurance_deposit` (DECIMAL) — сумма страхового депозита
- `working_deposit` (DECIMAL) — сумма рабочего депозита
- `insurance_deposit_confirmed` (BOOLEAN) — флаг подтверждения страхового депозита
- `is_active` (BOOLEAN) — активность аккаунта
- `created_at` (TIMESTAMP)

**2. tokens** — Токены авторизации
- `token_id` (SERIAL, PK)
- `token` (VARCHAR, UNIQUE) — 16-символьный ключ
- `role` (VARCHAR) — назначенная роль
- `created_by` (BIGINT, FK → users) — кто создал
- `is_active` (BOOLEAN)
- `created_at` (TIMESTAMP)

**3. payment_details** — Банковские реквизиты трейдеров
- `detail_id` (SERIAL, PK)
- `trader_id` (BIGINT, FK → users)
- `detail_name` (VARCHAR) — пользовательское название
- `detail_type` (VARCHAR) — card или sbp
- `bank_name` (VARCHAR)
- `full_name` (VARCHAR) — ФИО получателя
- `card_number` (VARCHAR)
- `phone_number` (VARCHAR)
- `min_amount` / `max_amount` (DECIMAL) — диапазон сумм
- `is_active` (BOOLEAN)
- `created_at` (TIMESTAMP)

**4. deals** — Сделки (заявки)
- `deal_id` (SERIAL, PK)
- `deal_number` (VARCHAR, UNIQUE) — человекочитаемый номер (DYYYYMMDD####)
- `operator_id` (BIGINT, FK → users)
- `trader_id` (BIGINT, FK → users)
- `payment_detail_id` (INT, FK → payment_details)
- `amount_rub` / `amount_usdt` (DECIMAL)
- `status` (VARCHAR) — pending, confirmed, expired, disputed, rejected
- `expires_at` (TIMESTAMP) — срок действия
- `confirmed_at` (TIMESTAMP)
- `dispute_opened` (BOOLEAN)
- `dispute_resolved` (BOOLEAN)
- `resolution` (VARCHAR)
- `is_deleted` (BOOLEAN) — мягкое удаление
- `created_at` (TIMESTAMP)

**5. exchange_rates** — История курсов
- `rate_id` (SERIAL, PK)
- `usdt_to_rub` (DECIMAL)
- `updated_by` (BIGINT, FK → users) — NULL при автообновлении
- `is_auto_updated` (BOOLEAN)
- `updated_at` (TIMESTAMP)

**6. disputes** — Спорные ситуации
- `dispute_id` (SERIAL, PK)
- `deal_id` (INT, FK → deals)
- `opened_by` (BIGINT, FK → users)
- `resolved_by` (BIGINT, FK → users)
- `opened_at` / `resolved_at` (TIMESTAMP)
- `resolution` (VARCHAR)
- `reason` (TEXT)

### 2.3.2. Физическая схема и связи

Отношения между сущностями:
- `users` 1:N `tokens` (владелец создаёт токены)
- `users` 1:N `payment_details` (трейдер владеет реквизитами)
- `users` 1:N `deals` (как operator_id и как trader_id — два FK)
- `deals` N:1 `payment_details` (каждая сделка привязана к одному реквизиту)
- `deals` 1:N `disputes` (одна сделка может иметь несколько записей спора)
- `users` 1:N `exchange_rates` (кто установил курс вручную)

Для обеспечения совместимости с уже существующими базами данных реализован механизм **мягкого удаления** (`is_deleted`) и **автоматического обновления структуры** таблиц (`update_tables_structure`). При запуске приложение проверяет наличие колонок и добавляет их, если они отсутствуют, что позволяет проводить zero-downtime миграции.

### 2.3.3. Примеры SQL-запросов

**Запрос на подбор реквизита для сделки** (используется в `DealService.create_deal`):
```sql
SELECT pd.*, u.telegram_id as trader_telegram_id, u.username,
       u.insurance_deposit_confirmed, u.is_active as trader_active
FROM payment_details pd
JOIN users u ON pd.trader_id = u.user_id
WHERE pd.is_active = TRUE
  AND $1 BETWEEN pd.min_amount AND pd.max_amount
  AND u.role = 'trader'
  AND u.insurance_deposit_confirmed = TRUE
  AND u.is_active = TRUE
ORDER BY RANDOM()
LIMIT 1
```

**Запрос статистики оператора:**
```sql
SELECT 
    COUNT(*) as total_deals,
    SUM(CASE WHEN status = 'confirmed' THEN 1 ELSE 0 END) as confirmed_deals,
    SUM(CASE WHEN status = 'confirmed' THEN amount_rub ELSE 0 END) as total_rub
FROM deals 
WHERE operator_id = $1 AND is_deleted = FALSE
```

## 2.6. Машина состояний для многошаговых сценариев

### 2.6.1. Концепция FSM в aiogram

Для реализации сценариев, требующих пошагового ввода данных от пользователя (добавление реквизита, установка курса, поиск по датам), используется механизм **Finite State Machine (FSM)** — конечный автомат состояний.

Каждый многошаговый сценарий описывается классом-наследником `StatesGroup`, где каждый атрибут — это отдельное состояние (State). Переход между состояниями осуществляется методами `state.set_state()` и `state.update_data()`.

### 2.6.2. Сценарий «Добавление реквизита трейдером»

```
[ожидание_названия] ──название──→ [ожидание_типа]
      ↑                                 │
      └──────────────назад──────────────┘

[ожидание_типа] ──тип──→ [ожидание_банка]
      ↑                        │
      └──────────назад─────────┘

[ожидание_банка] ──банк──→ [ожидание_ФИО]
      ↑                          │
      └──────────назад───────────┘

[ожидание_ФИО] ──ФИО──→ [ожидание_карты] (если тип=card)
      │                      │
      │                      └──→ [ожидание_телефона] (если тип=sbp)
      │                             │
      └──────────назад──────────────┘

[ожидание_карты/телефона] ──номер──→ [ожидание_мин_суммы]
      │                                  │
      └──────────────назад───────────────┘

[ожидание_мин_суммы] ──сумма──→ [ожидание_макс_суммы]
      │                              │
      └──────────────назад───────────┘

[ожидание_макс_суммы] ──сумма──→ [сохранение_в_БД] ──→ Главное меню
```

На каждом шаге система валидирует ввод (длина, формат, диапазоны) и предоставляет кнопку «Назад» для возврата к предыдущему состоянию без потери уже введённых данных (данные сохраняются в FSM-хранилище).

### 2.6.3. Другие сценарии с FSM

- **Создание заявки оператором**: ожидание_суммы → валидация → создание сделки.
- **Установка курса владельцем**: ожидание_курса → валидация (число > 0) → сохранение.
- **Поиск транзакций**: ожидание_запроса → поиск по номеру/сумме → вывод результатов.
- **Удаление сделки**: ожидание_номера → проверка существования → мягкое удаление.
- **Период статистики**: ожидание_начальной_даты → ожидание_конечной_даты → расчёт.

## 3.2. Реализация модуля авторизации и аутентификации

### 3.2.1. Принцип работы токен-авторизации

В отличие от классической схемы «логин + пароль», система использует **разовые токены** (invitation tokens). Владелец генерирует токен для конкретной роли и передаёт его сотруднику. Сотрудник вводит `/token <ключ>`, и система:
1. Проверяет существование токена в таблице `tokens` с `is_active = TRUE`.
2. Привязывает Telegram ID пользователя к найденной роли.
3. Создаёт или обновляет запись в таблице `users`.
4. Для трейдеров без подтверждённого депозита устанавливает `is_active = FALSE`.

Эта схема удобна для закрытых команд: не требует хранения паролей, позволяет мгновенно отзывать доступ деактивацией токена, предотвращает саморегистрацию.

### 3.2.2. Класс AuthService

Модуль `services/auth_service.py` реализует чистый сервисный класс со статическими методами:

- `authenticate_user(token, telegram_id, username)` — полный цикл авторизации с логированием каждого шага.
- `logout_user(telegram_id)` — очистка поля `token` у пользователя.
- `get_user_data(telegram_id)` — получение полного профиля.
- `can_trader_access(telegram_id)` — проверка готовности трейдера к работе.
- `activate_trader(trader_id)` — активация после подтверждения депозита владельцем.

Логирование на уровне `INFO` позволяет отслеживать попытки авторизации в продакшене без раскрытия чувствительных данных.

## 3.3. Реализация ролевой модели пользователей

### 3.3.1. Главное меню

Интерфейс системы адаптируется под роль пользователя. Функция `get_main_menu(role, deposit_confirmed)` из `keyboards/menus.py` возвращает различные наборы кнопок:

- **Трейдер без депозита**: только «Пополнить депозит» и «Выйти» — минимальный доступ.
- **Трейдер с депозитом**: Личный кабинет, Реквизиты, Сделки, Активные заявки, Споры, Пополнить депозит, Выйти.
- **Оператор**: Создать заявку, Активные заявки, Статистика, Поиск транзакций, Выйти.
- **Владелец**: Общая статистика, Управление токенами, Курс USDT, Пользователи, Автообновление, Удалить сделку, Выйти.

### 3.3.2. Регистрация обработчиков

В `app.py` регистрация производится через Router:

```python
router = Router()
```

Каждый модуль `handlers/*.py` экспортирует функцию `register_*_handlers(router)`, которая связывает текстовые команды, callback-запросы и FSM-состояния с соответствующими асинхронными функциями.

## 3.4. Реализация модуля управления сделками

### 3.4.1. Жизненный цикл сделки

Модуль `services/deal_service.py` инкапсулирует всю логику работы со сделками:

**Создание сделки (`create_deal`):**
1. Получает сумму в RUB от оператора.
2. Запрашивает текущий курс из `ExchangeService`.
3. Вычисляет `amount_usdt = round(amount_rub / rate, 2)`.
4. Выполняет SQL-запрос с `ORDER BY RANDOM()` для справедливого распределения нагрузки между трейдерами.
5. Генерирует уникальный номер: `D{YYYYMMDD}{random 4 digits}`.
6. Устанавливает `expires_at = NOW() + 30 минут`.
7. Сохраняет запись в `deals` со статусом `pending`.

**Автоматическое архивирование (`check_expired_deals`):**
Планировщик каждые 5 минут запускает проверку:
```sql
UPDATE deals 
SET status = 'expired'
WHERE status = 'pending' 
  AND expires_at < NOW()
  AND (is_deleted = FALSE OR is_deleted IS NULL)
RETURNING *
```

Это гарантирует, что неоплаченные заявки не висят в системе вечно.

**Подтверждение (`confirm_deal`):**
Трейдер нажимает кнопку «Подтвердить» → обновляется статус на `confirmed`, заполняется `confirmed_at`, `dispute_resolved = TRUE`, `resolution = 'confirmed'`.

**Мягкое удаление (`delete_deal`):**
Вместо физического `DELETE` используется `UPDATE deals SET is_deleted = TRUE`, что сохраняет аудиторскую следопытность.

### 3.4.2. Подбор реквизитов

Алгоритм подбора реквизитов учитывает комплекс условий:
- Реквизит активен (`is_active = TRUE`).
- Сумма попадает в диапазон `[min_amount, max_amount]`.
- Трейдер имеет подтверждённый страховой депозит.
- Аккаунт трейдера активен.
- Случайный порядок (`RANDOM()`) для равномерного распределения заявок.

Если подходящих реквизитов нет, оператор получает сообщение: «Не найдено подходящих реквизитов для этой суммы».

## 3.5. Реализация модуля парсинга курса USDT/RUB

### 3.5.1. Постановка задачи

Актуальный курс обмена USDT к рублю — один из ключевых бизнес-параметров. Система должна:
- Получать курс автоматически, без участия человека.
- Использовать несколько независимых источников для повышения надёжности.
- Валидировать полученное значение (защита от аномалий и ошибок API).
- При полном отказе внешних сервисов переходить на резервное значение.
- Сохранять историю изменений для аудита.

### 3.5.2. Архитектура модуля

Модуль `services/exchange_service.py` реализует паттерн **«Цепочка ответственности»** (Chain of Responsibility): система перебирает источники по приоритету до первого успешного ответа.

Приоритет источников:
1. **Binance P2P** — крупнейшая P2P-площадка, наиболее актуальный рыночный курс.
2. **KuCoin P2P** — альтернативная площадка, резервный источник.
3. **ЦБ РФ (USD) + 2%** — официальный курс доллара с премией за USDT.

Если все источники недоступны — используется резервный курс `78.50 RUB/USDT`.

### 3.5.3. Валидация и защита

Полученный курс проходит проверку:
```python
if rate and 50 <= rate <= 200:
    await ExchangeService._save_rate(rate, None, True)
    return rate
```

Диапазон 50–200 RUB/USDT выбран на основе исторических данных: за последние 3 года курс USDT/RUB колебался в пределах 60–140. Значения за пределами диапазона считаются аномальными (ошибка API, тестовые данные, DDOS-защита) и отбрасываются.

### 3.5.4. Интеграция с планировщиком

В `app.py` планировщик APScheduler настроен на запуск каждый час:
```python
scheduler.add_job(scheduled_tasks, 'interval', minutes=60)
```

Функция `scheduled_tasks` вызывает `DealService.check_expired_deals()` и `ExchangeService.update_rate_automatically()`, обеспечивая актуальность данных без ручного вмешательства.

### 3.5.5. Сохранение курса

Каждое обновление курса записывается в таблицу `exchange_rates`:
```sql
INSERT INTO exchange_rates (usdt_to_rub, updated_by, is_auto_updated)
VALUES ($1, $2, $3)
```

Поле `updated_by` заполняется только при ручной установке курса владельцем. При автообновлении оно равно `NULL`, а `is_auto_updated = TRUE`. Это позволяет в дальнейшем анализировать, насколько автоматический курс отличается от ручных корректировок.

## 3.6. Реализация подсистемы управления реквизитами

### 3.6.1. Многошаговый ввод

Добавление реквизита реализовано через FSM-состояния (`TraderStates`) в `handlers/trader.py`. Пользователь проходит 7 шагов:
1. Ввод названия (например, «Тинькофф основная»).
2. Выбор типа: карта или СБП (кнопки ReplyKeyboard).
3. Ввод названия банка.
4. Ввод ФИО получателя (минимум 2 слова).
5. Ввод номера карты (16–19 цифр) или телефона (10–11 цифр).
6. Ввод минимальной суммы (RUB).
7. Ввод максимальной суммы (должна быть > минимальной).

На каждом шаге выполняется валидация: проверка длины, формата, числового диапазона. При ошибке пользователь получает конкретное сообщение («Номер карты должен содержать 16–19 цифр») и остаётся в текущем состоянии.

### 3.6.2. Управление реквизитами

В разделе «Мои реквизиты» трейдер видит список всех своих записей с индикаторами статуса. Для каждого реквизита генерируется Inline-клавиатура с кнопками:
- «Включить/Выключить» — переключение `is_active`.
- «Удалить» — физическое удаление из БД (с проверкой на отсутствие активных сделок).

При переключении статуса сообщение редактируется через `edit_text`, что обеспечивает плавный UX без засорения чата новыми сообщениями.

## 3.7. Реализация системы споров и разрешения конфликтов

### 3.7.1. Открытие спора

Оператор в разделе «Активные заявки» видит Inline-кнопку «Открыть спор». При нажатии:
1. Статус сделки меняется на `disputed`.
2. Устанавливается `dispute_opened = TRUE`.
3. В таблицу `disputes` добавляется запись с `opened_by` = operator_id.

### 3.7.2. Участие трейдера

Трейдер в разделе «Споры» видит список спорных сделок с возможностью просмотра деталей. Трейдер не может закрыть спор самостоятельно — это исключительная компетенция владельца.

### 3.7.3. Разрешение владельцем

Владелец имеет доступ к полной статистике и может просматривать все спорные сделки. Разрешение спора предполагает выбор исхода (`confirmed` или `rejected`) и обновление записей в таблицах `deals` и `disputes`. В текущей реализации механизм разрешения доступен через прямое взаимодействие с БД, а интерфейс владельца расширяется по мере необходимости.

## 3.8. Реализация депозитной подсистемы

### 3.8.1. Страховой депозит

Каждый новый трейдер при авторизации получает `is_active = FALSE` и `insurance_deposit_confirmed = FALSE`. В приветственном сообщении ему отправляется:
- Сумма депозита: 1000 USDT.
- Адрес кошелька владельца (USDT TRC-20).
- Инструкция обратиться к владельцу после перевода.

После реального перевода владелец вручную подтверждает депозит через меню «Управление пользователями» → «Подтвердить страховой депозит». При подтверждении:
1. Устанавливаются `insurance_deposit_confirmed = TRUE`, `is_active = TRUE`.
2. Трейдеру отправляется уведомление через `bot.send_message`.
3. В главном меню трейдера появляются полноценные функции.

### 3.8.2. Рабочий депозит

Трейдер может пополнять рабочий депозит через меню «Пополнить депозит». Система запрашивает сумму, проверяет её корректность и добавляет к полю `working_deposit`. Владелец впоследствии подтверждает поступление через аналогичный механизм.

## 3.9. Реализация сервиса статистики и аналитики

### 3.9.1. Статистика владельца

Меню «Общая статистика» предоставляет сводку:
- Количество пользователей (всего, трейдеров, операторов).
- Финансовые показатели: количество сделок, общий оборот в RUB и USDT, средняя сумма сделки.
- Распределение по статусам: подтверждено, в ожидании, истекло, споры, отклонено.
- Текущий курс USDT/RUB.

Владелец может задать произвольный период (начальная и конечная дата), и система строит отчёт с фильтром `created_at BETWEEN start AND end`.

### 3.9.2. Статистика оператора и трейдера

Оператор видит только свои заявки (`operator_id = $1`). Трейдер видит данные по своим сделкам (`trader_id = $1`) и реквизитам. Это обеспечивает конфиденциальность данных в многопользовательской среде.

## 3.10. Пользовательский интерфейс чат-бота

### 3.10.1. Reply-клавиатуры

Основной способ взаимодействия — постоянная Reply-клавиатура, которая отображается под полем ввода сообщения. Это удобно для мобильных устройств: пользователь видит все доступные действия и не запоминает текстовые команды.

Клавиатуры адаптируются под контекст:
- Главное меню — полный набор ролевых функций.
- Меню реквизитов — добавление, список, назад.
- Меню токенов — создание, список, деактивация, назад.
- Кнопка «Назад» — единичная клавиатура для возврата в предыдущее состояние FSM.

### 3.10.2. Inline-клавиатуры

Inline-кнопки используются для действий внутри конкретного сообщения:
- Подтверждение/отклонение сделки (`confirm_123`, `reject_123`).
- Включение/выключение реквизита (`toggle_detail_456`).
- Удаление реквизита (`delete_detail_456`).
- Открытие спора (`open_dispute_789`).
- Обновление курса (`update_rate_auto`, `set_rate_manual`).

Callback-данные формируются по шаблону `<action>_<entity_id>`, что позволяет одному обработчику обслуживать множество объектов.

### 3.10.3. UX-паттерны

- **Эмодзи-индикаторы**: ⏳ — в ожидании, ✅ — подтверждено, ❌ — истекло/отклонено, ⚖️ — спор, 🟢 — активен, 🔴 — неактивен.
- **Форматирование Markdown**: жирный текст для выделения ключевых сумм и адресов, моноширинный текст для токенов и номеров кошельков.
- **Редактирование сообщений**: при изменении статуса реквизита или сделки старое сообщение обновляется вместо отправки нового.
- **Уведомления**: при подтверждении депозита трейдер получает прямое сообщение от бота, даже если не взаимодействовал с системой в данный момент.

## 3.11. Развёртывание системы на облачной платформе

### 3.11.1. Платформа Render

Render — это PaaS-платформа, предоставляющая бесплатный тариф для веб-сервисов. Выбрана по причинам:
- Нативная поддержка Python-приложений.
- Автоматический HTTPS.
- Простая интеграция с GitHub.
- Постоянный URL для webhook.

### 3.11.2. Конфигурация деплоя

Файл `render.yaml` описывает сервис:
```yaml
services:
  - type: web
    name: trade-prog-2
    env: python
    plan: free
    buildCommand: pip install -r requirements.txt
    startCommand: python app.py
    envVars:
      - key: PYTHON_VERSION
        value: 3.11.0
```

Файл `Procfile` (альтернативная конфигурация для Heroku-совместимости):
```
web: python app.py
```

### 3.11.3. Webhook-сервер

При запуске на Render приложение:
1. Создаёт `aiohttp.web.Application()`.
2. Регистрирует эндпоинт `/health` для keep-alive и мониторинга.
3. Удаляет старый webhook и устанавливает новый на URL `https://trade-prog-2.onrender.com/webhook`.
4. Регистрирует `SimpleRequestHandler` для маршрутизации Telegram-обновлений в aiogram Dispatcher.
5. Запускает TCP-сервер на порту из переменной `PORT` (по умолчанию 8080).
6. Запускает фоновую задачу `keep_alive_ping()`, которая каждые 10 минут обращается к `/health`.

### 3.11.4. База данных Neon

В качестве облачной СУБД используется Neon — serverless PostgreSQL с бесплатным тарифом. Подключение осуществляется через `DATABASE_URL`, содержащую все параметры аутентификации. Пул соединений `asyncpg` автоматически управляет жизненным циклом TCP-соединений к удалённому серверу.

## 3.12. Функциональное тестирование

### 3.12.1. Методология тестирования

Тестирование проводилось методом **ручного функционального тестирования** (manual black-box testing) по ключевым бизнес-сценариям. Каждый сценарий включал:
- Предусловие (состояние системы перед тестом).
- Шаги воспроизведения.
- Ожидаемый результат.
- Фактический результат.
- Статус (пройден / не пройден).

### 3.12.2. Тест-кейсы

**ТК-1: Авторизация нового пользователя**
- Предусловие: в системе есть активный токен `operator`.
- Шаги: новый пользователь отправляет `/start`, затем `/token <ключ>`.
- Ожидаемый результат: пользователь авторизован, получает меню оператора.
- Фактический результат: соответствует ожидаемому.
- Статус: **Пройден**.

**ТК-2: Создание заявки оператором**
- Предусловие: авторизован оператор, в системе есть активный трейдер с реквизитами.
- Шаги: нажать «Создать заявку» → ввести 15000 → получить реквизиты.
- Ожидаемый результат: сделка создана, номер присвоен, сумма в USDT рассчитана по текущему курсу.

# ПРИЛОЖЕНИЕ А. ПОЛНЫЙ ЛИСТИНГ МОДУЛЯ ПАРСИНГА КУРСА USDT/RUB

**Файл:** `services/exchange_service.py`

```python
import aiohttp
from bs4 import BeautifulSoup
import re
from datetime import datetime
from database.models import db
from config.settings import config

class ExchangeService:
    @staticmethod
    async def get_current_rate():
        """Get current USDT to RUB rate"""
        async with db.pool.acquire() as conn:
            rate = await conn.fetchrow(
                "SELECT usdt_to_rub FROM exchange_rates ORDER BY updated_at DESC LIMIT 1"
            )
            return float(rate['usdt_to_rub']) if rate else 90.0
    
    @staticmethod
    async def update_rate_automatically():
        """Auto-update rate from multiple sources"""
        try:
            print("🔄 Получаю курс USDT/RUB...")

            # Try multiple sources in order
            sources = [
                ExchangeService._parse_binance,
                ExchangeService._parse_kucoin,
                ExchangeService._parse_cbr
            ]

            for source in sources:
                try:
                    rate = await source()
                    if rate and 50 <= rate <= 200:
                        await ExchangeService._save_rate(rate, None, True)
                        print(f"✅ Курс USDT/RUB успешно обновлен: {rate}")
                        return rate
                except Exception as e:
                    print(f"⚠️ Источник {source.__name__} не сработал: {e}")
                    continue

            print("❌ Не удалось получить курс ни из одного источника, используем резервный курс")
            # Use fallback rate of 78.50
            await ExchangeService._save_rate(78.50, None, True)
            print(f"✅ Установлен резервный курс: 78.50")
            return 78.50

        except Exception as e:
            print(f"❌ Общая ошибка: {e}")
            # Use fallback rate
            await ExchangeService._save_rate(78.50, None, True)
            print(f"✅ Установлен резервный курс: 78.50")
            return 78.50

    @staticmethod
    async def _parse_binance():
        """Parse rate from Binance P2P"""
        headers = {
            'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36'
        }
        async with aiohttp.ClientSession() as session:
            async with session.get('https://p2p.binance.com/bapi/c2c/v2/friendly/c2c/adv/search',
                                   headers=headers, timeout=10,
                                   params={
                                       'fiat': 'RUB',
                                       'asset': 'USDT',
                                       'tradeType': 'BUY',
                                       'page': 1,
                                       'rows': 1
                                   }) as response:
                if response.status == 200:
                    data = await response.json()
                    if data.get('data') and len(data['data']) > 0:
                        rate = float(data['data'][0]['adv']['price'])
                        print(f"✅ Курс с Binance: {rate}")
                        return rate
        return None

    @staticmethod
    async def _parse_kucoin():
        """Parse rate from KuCoin P2P"""
        headers = {
            'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36'
        }
        async with aiohttp.ClientSession() as session:
            async with session.get('https://www.kucoin.com/_api/p2p/market/advertisement/list',
                                   headers=headers, timeout=10,
                                   params={
                                       'currency': 'USDT',
                                       'fiat': 'RUB',
                                       'tradeType': 'BUY',
                                       'page': 1,
                                       'pageSize': 1
                                   }) as response:
                if response.status == 200:
                    data = await response.json()
                    if data.get('data') and data['data'].get('items') and len(data['data']['items']) > 0:
                        rate = float(data['data']['items'][0]['price'])
                        print(f"✅ Курс с KuCoin: {rate}")
                        return rate
        return None

    @staticmethod
    async def _parse_bybit():
        """Parse rate from Bybit P2P"""
        headers = {
            'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36'
        }
        async with aiohttp.ClientSession() as session:
            async with session.get('https://api.bybit.com/v5/market/tickers?category=spot&symbol=USDTUSDT',
                                   headers=headers, timeout=10) as response:
                if response.status == 200:
                    data = await response.json()
                    if data.get('retCode') == 0 and data.get('result') and data['result'].get('list'):
                        # Bybit shows USDT/USDT, we need to estimate RUB rate
                        # This is a fallback - use a fixed multiplier
                        usdt_usdt = float(data['result']['list'][0]['lastPrice'])
                        # Use a reasonable RUB rate as fallback
                        rate = 90.0  # Fallback rate
                        print(f"✅ Курс с Bybit (fallback): {rate}")
                        return rate
        return None

    @staticmethod
    async def _parse_cbr():
        """Parse rate from Central Bank of Russia (USD to RUB)"""
        try:
            headers = {
                'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/91.0.4472.124 Safari/537.36',
                'Accept': 'application/json'
            }
            async with aiohttp.ClientSession() as session:
                async with session.get('https://www.cbr-xml-daily.ru/daily_json.js',
                                       headers=headers, timeout=10, skip_auto_headers=['Accept']) as response:
                    if response.status == 200:
                        text = await response.text()
                        # Parse JSON from text
                        import json
                        data = json.loads(text)
                        if data.get('Valute') and 'USD' in data['Valute']:
                            # Get USD rate from CBR and add premium for USDT
                            usd_rate = float(data['Valute']['USD']['Value'])
                            # USDT typically trades at a premium to USD in Russia
                            rate = usd_rate * 1.02  # Add 2% premium
                            print(f"✅ Курс с ЦБ РФ (USD + премия): {rate}")
                            return rate
        except Exception as e:
            print(f"⚠️ ЦБ РФ не сработал: {e}")
        return None
    
    @staticmethod
    async def _save_rate(rate: float, user_id: int = None, auto: bool = True):
        """Save rate to database"""
        async with db.pool.acquire() as conn:
            await conn.execute(
                '''
                INSERT INTO exchange_rates (usdt_to_rub, updated_by, is_auto_updated)
                VALUES ($1, $2, $3)
                ''',
                rate, user_id, auto
            )
    
    @staticmethod
    async def set_manual_rate(rate: float, user_id: int):
        """Set rate manually by owner"""
        await ExchangeService._save_rate(rate, user_id, False)
        print(f"✅ Курс установлен вручную: {rate} RUB")
        return rate
    
    @staticmethod
    async def clear_old_rates():
        """Очистить старые записи о курсах (оставить последние 100)"""
        async with db.pool.acquire() as conn:
            result = await conn.execute('''
                DELETE FROM exchange_rates 
                WHERE rate_id NOT IN (
                    SELECT rate_id FROM exchange_rates 
                    ORDER BY updated_at DESC 
                    LIMIT 100
                )
            ''')
            print(f"✅ Старые курсы очищены")
```
